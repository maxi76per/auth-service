# Объяснение кода простыми словами

Разберём очень простым языком. У вас этот файл `C:\Users\User\auth-service\index.js` делает маленький сервер для входа пользователей.

## Общая идея

Сервер умеет:
- регистрировать пользователя
- проверять логин и пароль
- выдавать токен
- пускать или не пускать в защищённый маршрут `/admin`
- блокировать вход после 3 ошибок
- хранить только одну активную сессию

Можно представить так:

1. Пользователь отправляет логин и пароль.
2. Сервер проверяет их.
3. Если всё верно, сервер выдаёт JWT-токен.
4. Потом этот токен отправляется в защищённые запросы.
5. Сервер по токену понимает, кто это и можно ли ему дать доступ.

## Что подключено в начале файла

В начале файла:

- `express` — чтобы создать веб-сервер
- `bcryptjs` — чтобы хешировать пароли
- `crypto` — чтобы создавать уникальный `sessionId`
- `jsonwebtoken` — чтобы создавать и проверять JWT

Это строки в `index.js`.

## Что такое `app`

В начале создаётся приложение:

```js
const app = express();
app.use(express.json());
```

Это значит:
- `app` — ваш сервер
- `express.json()` говорит серверу: “если клиент прислал JSON, распарсь его в `req.body`”

То есть когда вы отправляете:

```json
{"username":"ivan","password":"123456"}
```

сервер сможет прочитать это из `req.body`.

## Настройки

Есть настройки:

- `PORT` — на каком порту работает сервер
- `JWT_SECRET` — секретный ключ для подписи токенов
- `MAX_LOGIN_ATTEMPTS = 3` — максимум 3 ошибки
- `LOCK_TIME_MS = 60 * 1000` — блокировка на 1 минуту

## Где хранятся пользователи

В коде есть:

```js
let users = [
  createUser(1, 'admin', 'admin123', 'admin'),
  createUser(2, 'moderator', 'moderator123', 'moderator')
];
```

Это временное хранилище в памяти.

Важно понимать:
- это не настоящая база данных
- после перезапуска сервера все зарегистрированные вручную пользователи исчезнут
- останутся только те, которые заново создаются в коде, например `admin`

## Как устроен пользователь

У пользователя есть поля:

- `id` — номер пользователя
- `username` — логин
- `passwordHash` — не пароль, а его зашифрованная версия
- `role` — роль (`user`, `moderator`, `admin`)
- `lastLogin` — время последнего входа
- `activeToken` — текущий действующий токен
- `failedLoginAttempts` — сколько раз подряд ошиблись паролем
- `lockUntil` — до какого времени аккаунт заблокирован

## Почему пароль не хранится как обычный текст

При регистрации используется:

```js
const passwordHash = await bcrypt.hash(password, 10);
```

Это значит:
- пользователь вводит обычный пароль
- сервер превращает его в хеш
- сохраняется только хеш

Это правильно и безопаснее, чем хранить пароль как есть.

## Маршрут `/register` — регистрация

Когда вы отправляете POST-запрос на `/register`, сервер:

1. Берёт `username`, `password`, `role`
2. Проверяет, что логин и пароль вообще пришли
3. Проверяет, что роль допустимая
4. Проверяет, нет ли уже такого пользователя
5. Хеширует пароль через `bcrypt`
6. Создаёт объект пользователя
7. Кладёт его в массив `users`
8. Возвращает ответ, что регистрация прошла успешно

Проще говоря: `/register` добавляет нового пользователя в систему.

## Маршрут `/login` — вход

Когда вы отправляете POST на `/login`, сервер делает вот что:

1. Проверяет, пришли ли логин и пароль
2. Ищет пользователя по `username`
3. Если пользователя нет — возвращает `401`
4. Проверяет, не заблокирован ли аккаунт
5. Сравнивает введённый пароль с хешем через `bcrypt.compare`
6. Если пароль неверный:
   - увеличивает счётчик ошибок
   - после 3 ошибок ставит блокировку на 1 минуту
   - возвращает `401`
7. Если пароль верный:
   - сбрасывает счётчик ошибок
   - снимает блокировку
   - создаёт JWT
   - сохраняет этот токен как `activeToken`
   - пишет лог в консоль
   - возвращает токен клиенту

## Как работает блокировка после 3 ошибок

Если пароль неверный:

```js
user.failedLoginAttempts += 1;
```

То есть плюс одна ошибка.

Когда ошибок становится 3:

```js
user.lockUntil = Date.now() + LOCK_TIME_MS;
user.failedLoginAttempts = 0;
```

Это значит:
- мы запоминаем время, до которого вход запрещён
- ошибки обнуляем

Потом при следующем входе сервер сравнивает:

```js
if (user.lockUntil && user.lockUntil > Date.now())
```

Если время блокировки ещё не прошло, сервер не даёт войти.

## Как создаётся JWT

Токен создаётся так:

```js
const token = jwt.sign(
  { userId: user.id, role: user.role, sessionId: crypto.randomUUID() },
  JWT_SECRET,
  { expiresIn: '15m' }
);
```

Что здесь происходит:
- в токен записывается `userId`
- в токен записывается `role`
- в токен записывается уникальный `sessionId`
- токен подписывается секретным ключом
- срок жизни токена — 15 минут

То есть токен — это как временный цифровой пропуск.

## Зачем нужен `sessionId`

Чтобы новая сессия отличалась от старой.

Если пользователь вошёл ещё раз, создаётся новый токен с новым `sessionId`.
Потом в `activeToken` сохраняется только последний токен.

Из-за этого старый токен перестаёт работать.

Это и есть “одна активная сессия”.

## Как работает `authenticate`

Функция `authenticate` — это middleware, то есть промежуточная проверка перед доступом к защищённым маршрутам.

Она делает так:

1. Берёт заголовок `Authorization`
2. Проверяет, есть ли токен вообще
3. Достаёт токен после слова `Bearer`
4. Проверяет JWT через `jwt.verify`
5. Находит пользователя по `userId`
6. Проверяет, совпадает ли присланный токен с `activeToken`
7. Если всё хорошо — пропускает дальше
8. Если нет — возвращает `401`

Именно здесь обеспечивается:
- токен настоящий
- токен не истёк
- токен именно текущей активной сессии

## Как работает `/admin`

Маршрут `/admin` защищён в два шага:

1. Сначала срабатывает `authenticate`
   - без токена не пустит
   - с неправильным токеном не пустит
2. Потом идёт проверка роли
   - если роль не `admin`, ответ `403 Forbidden`

То есть:
- `401` — не прошёл проверку личности
- `403` — личность известна, но прав недостаточно

Это полезно запомнить для защиты.

## Что такое логирование входов

В коде есть функция:

```js
function logAttempt(username, success) {
  console.log({
    username,
    success,
    time: new Date()
  });
}
```

Это просто вывод в консоль:
- кто пытался войти
- успешно или нет
- когда это произошло

Когда вы смотрите на окно, где запущен сервер, эти записи появляются там.

## Как сервер запускается

Внизу файла:

```js
app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

Это означает:
- сервер начинает слушать порт
- после запуска пишет сообщение в консоль

## Как это всё связать в голове

Самая простая схема такая:

1. `register`
   - создаёт пользователя
   - пароль превращает в хеш

2. `login`
   - ищет пользователя
   - сравнивает пароль с хешем
   - если верно, выдаёт JWT
   - если 3 ошибки, блокирует на минуту

3. `authenticate`
   - проверяет токен
   - проверяет, что это активная сессия

4. `/admin`
   - после проверки токена смотрит роль
   - пропускает только `admin`



“В памяти хранится массив пользователей. При регистрации пароль хешируется через bcrypt. При логине пароль сравнивается с хешем. Если логин успешный, сервер выдаёт JWT на 15 минут. Для защищённых маршрутов токен проверяется middleware-функцией authenticate. В токене есть роль, поэтому на `/admin` дополнительно проверяется, что роль равна `admin`. После трёх ошибок вход блокируется на минуту. Также хранится только один activeToken, поэтому новый вход делает старую сессию невалидной.”
